## 3. Data Synchronization Architecture

### 3.1 How Sync Works

```
┌─────────────┐         ┌──────────────────────┐         ┌──────────────────────┐
│  MSSQL DB   │ ──────> │  Sync Service         │ ──────> │  PostgreSQL (Neon)   │
│ (ASP.NET)   │  read   │  (periodic + webhook) │  write  │  synced_* tables     │
└─────────────┘         └──────────────────────┘         └──────────────────────┘
```

### 3.2 Sync Triggers
- **Initial full sync**: On first deployment
- **Periodic sync**: Every 5 minutes via background task
- **Webhook sync**: ASP.NET backend calls `POST /sync/refresh` when data changes
- **Incremental sync**: Only fetch rows where `LastModifiedAt > last_synced_at`

## 5. Scoring System

### 5.1 Property Recommendation Score

```
Total Score = Σ(weight_i × score_i)  /  Σ(weight_i)

where:
  w_budget × score_budget     (30%)  — how well monthly_rent fits user's budget range
  w_location × score_location (25%)  — distance from preferred city/governorate
  w_amenities × score_amenity (15%)  — matching requested amenities
  w_tenant × score_tenant     (10%)  — gender/student/worker compatibility
  w_furnished × score_furnished (5%) — furnished status match
  w_type × score_type         (10%)  — full/shared/room preference match
  w_recent × score_recent     (5%)   — recently added properties get boost
```

### 5.2 Score Breakdown Details

#### Budget Scorer
```
score_budget(user, property):
  IF monthly_rent IS NULL:
    return 0.5                                  # no price = neutral
  IF monthly_rent <= max_budget AND >= min_budget:
    return 1.0                                   # perfect fit
  IF monthly_rent <= max_budget * 1.2:
    return 0.7                                   # slightly over budget
  IF min_budget IS NULL AND monthly_rent <= max_budget:
    return 0.9                                   # no min set
  ELSE:
    return max(0, 1 - (monthly_rent - max_budget) / max_budget)  # degrade linearly
```

#### Location Scorer
```
score_location(user, property):
  IF user.city == property.city OR user.governorate == property.governorate:
    return 1.0
  IF user.governorate AND property.governorate:
    distance = geo_distance(user.governorate_center, property.governorate_center)
    return max(0, 1 - distance / 200)  # 200km = zero score
  IF no location preference:
    return 0.5  # neutral
  ELSE:
    return 0.0
```

#### Tenant Scorer
```
score_tenant(user, property, allowed_tenants):
  IF user.gender == male AND allowed_tenants.StudentGender == 0:
    return 1.0
  IF user.gender == female AND allowed_tenants.StudentGender == 1:
    return 1.0
  IF allowed_tenants.AllowsStudents AND user.occupation == student:
    return 1.0
  IF allowed_tenants.AllowsWorkers AND user.occupation == worker:
    return 1.0
  IF no tenant restrictions:
    return 0.8  # flexible
  ELSE:
    return 0.0  # blocked
```

### 5.3 Room Recommendation Score

```
Same as property but with additional:
```

## 6. Matching Engine (Secret Roommate Compatibility)

### 6.1 Pairwise Compatibility Calculation

```
compatibility(A, B) = Σ(w_i × similarity_i)  /  Σ(w_i)

where:
  w_questionnaire × similarity_questionnaire   (50%)  — questionnaire answer overlap
  w_gender × similarity_gender                  (15%)  — same gender match
  w_occupation × similarity_occupation          (10%)  — both students or both workers
  w_age_group × similarity_age                  (10%)  — close age groups
  w_lifestyle × similarity_lifestyle            (15%)  — questionnaire lifestyle questions
```

### 6.2 Questionnaire Similarity

```python
def questionnaire_similarity(user_a_answers: dict, user_b_answers: dict) -> float:
    """
    Compare pairwise answers from the same questions.
    
    For scale_1_5 questions:
        similarity = 1 - (abs(a - b) / 4)   # 1-5 scale
    
    For multiple_choice questions:
        similarity = 1.0 if same choice else 0.0
    
    For boolean questions:
        similarity = 1.0 if same else 0.0
    
    Weighted average across all shared questions.
    """
```

### 6.3 Room-Level Score Aggregation

```python
def room_compatibility(seeker: User, room: Room, current_roommates: list[User]) -> float:
    """
    seeker ↔ roommate_A = 0.85
    seeker ↔ roommate_B = 0.62
    
    room_score = avg(0.85, 0.62) = 0.735
    
    adjust for:
    - room capacity available
    - number of roommates (more roommates = more complex)
    - gender balance in the room
    """
```

### 6.4 Privacy & Security

- **Roommate matching results are NEVER exposed via API to frontend**
- Only stored internally in `roommate_matches` table
- Frontend only sees the property/room recommendation, NOT individual roommate scores
- The score is used for internal ranking, not displayed to users
- API endpoint `/match/roommate` accepts seeker + room, returns score — consumed by ASP.NET backend only

## 7. Ranking Pipeline

```
User Request
    │
    ├─ 1. Load user profile + preferences + questionnaire answers
    ├─ 2. Load candidate pool (properties/rooms matching basic filters)
    │
    ├─ 3. Compute individual scores for each candidate
    │      ├─ BudgetScorer.score(user, candidate)
    │      ├─ LocationScorer.score(user, candidate)
    │      ├─ AmenityScorer.score(user, candidate)
    │      ├─ TenantScorer.score(user, candidate)
    │      └─ QuestionnaireScorer.score(user, candidate)  # if available
    │
    ├─ 4. Ranker.weighted_sum(scores, weights) → total_score
    ├─ 5. Sort by total_score DESC
    ├─ 6. Cache top N results in recommendation tables
    ├─ 7. Apply diversity boost (don't show 5 results from same building)
    └─ 8. Return ranked list
```

### Diversity Rule

When returning recommendations, if 3+ results are from the same property, keep the top 1 or 2 and replace the rest with the next-best from different properties. This prevents the user from seeing duplicate buildings.

## 8. Future ML Architecture (Extensibility)

The scoring system uses **Strategy Pattern** — each scorer is a separate class implementing `BaseScorer`:

```python
class BaseScorer(ABC):
    @abstractmethod
    def score(self, user, candidate, context) -> float: ...
```

This allows swapping in ML-based scorers later. Questionnaire answers are stored in our own PostgreSQL (not ASP.NET).

```
Phase 1 (Now):    Rule-based weighted scoring
Phase 2:          Collaborative filtering (user-user similarity)
Phase 3:          Content-based (property feature embeddings)
Phase 4:          Hybrid (weighted combination of all)
Phase 5:          Neural ranking model
```

### pgvector for Embeddings

The `property_embeddings` and `user_embeddings` tables use PostgreSQL `vector(384)` type via the `pgvector` extension. This enables:

```sql
-- Content-based: find properties similar to one the user liked
SELECT p.id, p.name, 1 - (p.embedding <=> ue.embedding) AS similarity
FROM property_embeddings p, user_embeddings ue
WHERE ue.user_id = :user_id
ORDER BY similarity DESC
LIMIT 20;

-- Collaborative: find users similar to current user
SELECT user_id, 1 - (embedding <=> :user_embedding) AS similarity
FROM user_embeddings
ORDER BY similarity DESC
LIMIT 10;
```

## 13. Final Architecture Decisions

| Question | Decision |
|---|---|
| **Neon DB** | Same DB as chatbot (existing Neon instance) |
| **User ID** | ASP.NET user_id (string) — passed by ASP.NET backend |
| **Questionnaire** | Owned by us — `questionnaire_categories`, `questions`, `answers` tables in our PostgreSQL. ASP.NET calls our `/questionnaire/*` APIs |
| **Sync** | Both: ASP.NET webhook `POST /sync/refresh` + periodic polling every 5 min |
| **pgvector** | Enable now — `CREATE EXTENSION vector;` in migration 006 |
| **Score cache** | No TTL — pre-computed in background, recomputed on data change |
| **API consumption** | Internal — only ASP.NET backend calls our endpoints, NOT frontend directly |
